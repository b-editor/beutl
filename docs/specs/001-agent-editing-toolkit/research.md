# Phase 0 Research: Agent Editing Toolkit

Six unknowns from the plan's Technical Context were investigated against the current codebase and the .NET ecosystem. Each is recorded as Decision / Rationale / Alternatives, with the concrete facts that ground the design. No `NEEDS CLARIFICATION` remained.

---

## 1. MCP hosting in .NET

**Decision**: Build the MCP server as a .NET console exe using the official **`ModelContextProtocol`** NuGet package (1.4.0, permissively licensed — Apache-2.0 per NuGet / MIT in the `modelcontextprotocol/csharp-sdk` repo, both permissive; maintained by Anthropic + Microsoft) with the Generic-Host model and **stdio** transport: `Host.CreateApplicationBuilder(args)` → `builder.Services.AddMcpServer().WithStdioServerTransport().WithTools<T>()` → `RunAsync()`. Tools are `[McpServerToolType]` classes with `[McpServerTool]` methods; `[Description]` on methods/parameters documents the tool and its input schema. Beutl services are injected as method parameters from the host DI container. `IMcpServer`, `CancellationToken`, `IProgress<>` are framework-injected. For `plan`/`apply`, return typed records so the SDK emits an output schema + structured content; for simple tools, return a serialized JSON string. Tools are registered explicitly so the public MCP surface can stay declarative even if internal convenience wrappers exist in the core assembly.

**Rationale**: The attribute + Generic-Host style is the SDK's documented happy path and maps onto Beutl's constructor-injection world with no manual JSON-RPC plumbing. Stdio is exactly what Claude Code's `.mcp.json` launcher spawns — no port, no auth, no ASP.NET Core. The package is **permissively licensed** (Apache-2.0 per NuGet / MIT in the repo — both permissive, no copyleft) and pulls only `Microsoft.Extensions.*`; nothing reaches the GPL worker. A permissive third-party SDK does not affect the MIT/GPL boundary, which governs first-party Beutl code and the GPL FFmpeg worker; confirm the exact license expression on the pinned package at implementation.

**Alternatives**: `ModelContextProtocol.AspNetCore` (HTTP/SSE) — rejected **for the standalone stdio server** (a host-spawned local exe needs no Kestrel/auth), but **selected for the in-app live-hosting endpoint** (§7) since a running GUI cannot be stdio-spawned. `ModelContextProtocol.Core` only — rejected, loses `WithTools<T>`/DI sugar. Assembly-wide registration was avoided for the final public surface because it exposes internal convenience wrappers. Hand-rolled JSON-RPC — rejected, re-implements framing/capability/schema the SDK maintains. Third-party alpha SDKs — rejected (unofficial).

**Key facts / gotchas**: latest stable `ModelContextProtocol` 1.4.0; also needs `Microsoft.Extensions.Hosting`. **CRITICAL**: route all logging to **STDERR** (`builder.Logging.AddConsole(o => o.LogToStandardErrorThreshold = LogLevel.Trace)`) — STDOUT is the JSON-RPC channel; any stray STDOUT write corrupts the protocol. `.mcp.json` stdio entry: `{ "command": "dotnet", "args": ["run","--project","<csproj>"], "env": { "BEUTL_WORKSPACE": "<root>" } }` (or the published exe path). Open risks: structured-output ergonomics (typed record vs serialized string) and net8/net9-SDK-TFM consumed from net10.0 — confirm at first build; pin the version; dispose Beutl services cleanly on stdin EOF.

## 2. Headless rendering & video export, and the GPL boundary

**Decision**: Both a still-frame PNG and a full audio+video export are feasible headlessly on the MIT side, with no Avalonia/window/UI thread. **Still**: `var r = new SceneRenderer(scene, RenderIntent.Delivery, renderScale: 1f, disableResourceShare: true, maxWorkingScale: float.PositiveInfinity, forceOriginalSource: true)` (note `SceneRenderer : Renderer`, exposing `Compositor`) → `var frame = r.Compositor.EvaluateGraphics(time + scene.Start)` → `r.Render(frame)` → `r.Snapshot()` (`Beutl.Media.Bitmap`; `Snapshot` verifies render-thread access, so run on `RenderThread.Dispatcher`) → `Bitmap.Save(path, EncodedImageFormat.Png)`. The delivery intent is mandatory, and final output forces original media rather than preview proxies. *(Corrected from an earlier `frame.Compositor.EvaluateGraphics` snippet — the compositor hangs off the renderer, not the frame; cf. `FrameProviderImpl.cs:49`.)* **Video**: replicate the orchestration in `OutputViewModel.StartEncode` inside a small headless host — build `SceneRenderer` + `FrameProviderImpl` (video) + `new SceneComposer(scene, RenderIntent.Delivery, disableResourceShare: true, forceOriginalSource: true)` + `SampleProviderImpl` (audio) and call `EncodingController.Encode(frameProvider, sampleProvider, ct)`. Both video and audio therefore use the same explicit delivery policy. Relocate `FrameProviderImpl`/`SampleProviderImpl` (currently in the Avalonia app `src/Beutl/Models`, but Avalonia-free) into `Beutl.Editor`.

**Rationale**: Rendering already runs on `RenderThread.Dispatcher` (a `Dispatcher.Spawn`'d background thread, no UI thread). The encode loop is plain async engine code; only progress `ReactiveProperty`/`NotificationService` are GUI-bound and a headless host omits them. The MIT→GPL reach is already correct: `FFmpegEncodingControllerProxy` spawns `Beutl.FFmpegWorker` as a **separate process** and talks to it only over `Beutl.FFmpegIpc` (named pipe + shared memory) — no `ProjectReference` to the worker. **Important correction**: that proxy lives in `src/Beutl.Extensions.FFmpeg/Encoding/` (not in `Beutl.FFmpegIpc`), and `Beutl.Extensions.FFmpeg.csproj` is MIT but `ProjectReference`s `Beutl.Controls` (Avalonia). So the headless toolkit must NOT compile-reference that extension. **Decision**: split the encode core (`FFmpegEncodingControllerProxy` + `FFmpegEncodingSettings` + their support types `FFmpegWorkerProcess`/`FFmpegWorkerLogPump`/`FFmpegWorkerCodecCache` and a UI-free libraries-missing state) into a MIT **non-UI encoder assembly** (`Beutl.Extensions.FFmpeg.Core`) the toolkit references directly. *(The AVFoundation encoder in `Beutl.Extensions.AVFoundation` is **already** non-UI MIT — its `.csproj` references only `Beutl.Extensibility`, no `Beutl.Controls` — so it needs **no split**: the toolkit references it directly, macOS-only. Verified against the repo; supersedes the earlier "plus AVFoundation equivalents [split]" wording.)* installed third-party encoders load via a **public headless encoder-registration service** factored out of the app-startup `LoadPrimitiveExtensionTask` / internal `PackageManager` (built-in registration is app-startup code today, so a headless path must be added). Everything is behind `Beutl.Extensibility.EncodingController`/`ControllableEncodingExtension` (encoder-agnostic), and `Beutl.FFmpegIpc` is reached only through the non-UI encoder assembly.

**Alternatives**: reference `src/Beutl` to reuse `OutputViewModel` — rejected (the app project is referenced only by `HeadlessUITests` by policy; `OutputViewModel` is `EditViewModel`-bound). Drive the live OutputTab UI — rejected (out of scope; the encode path needs no UI). Call FFmpeg/the worker directly — rejected (crosses the GPL boundary).

**Key facts / risks**: `RenderTarget.Create` uses the GPU (Vulkan; macOS = MoltenVK) when available and **falls back to a CPU SkiaSharp surface** when no device/Dispatcher (`GraphicsContextFactory.GetOrCreateShared()` returns null on failure). CPU fallback covers vector/text/bitmap stills **plus SKSL runtime-shader effects and particles** — both go through Skia's `SKRuntimeEffect.CreateShader` / `RenderTarget.Create`, which have CPU paths (verified `SKSLScriptEffect.cs:54`, `ParticleRenderNode.cs:209,263`, `RenderTarget.cs:56-74`); only **3D (Graphics3D/Vulkan) and any node whose render path hard-requires a GPU context** need a real GPU — surface a typed "rendering unavailable" error and gate those tests. FFmpeg **native libraries are a runtime prerequisite** (worker throws `FFmpegLibrariesNotFoundException`/exit 2) — preflight + typed error. Rendering must be marshaled onto `RenderThread.Dispatcher` (naive off-dispatcher calls hit `VerifyAccess`). Export supersampling/working-scale (`WorkingScaleCeiling.Export()` = +∞) is bounded by `ClampWorkingScaleToBufferBudget` — reuse the existing clamp.

## 3. Reconciliation: desired-state / merge-patch → undoable operations

**Decision**: Reconcile on Beutl's **observer-driven recording pipeline**, not by hand-constructing `ChangeOperation`s and not by `PopulateFromJsonObject` on the live tree. Per session, stand up the trio the editor uses (`EditViewModel.cs:117-127`): `var gen = new OperationSequenceGenerator(); var history = new HistoryManager(root, gen); var observer = new CoreObjectOperationObserver(null, root, gen); history.Subscribe(observer);`. To reconcile: (1) serialize the live root via `CoreSerializer.SerializeToJsonObject`; for a merge-patch request, apply the merge-patch to get the desired document (RFC 7396 for objects; **id-keyed merge for identity-bearing arrays** so a partial element/object/keyframe list does not drop unmentioned siblings — see §4); (2) structurally diff current-vs-desired into a leaf change list keyed by `CoreObject.Id` + property name; (3) inside `history.ExecuteInTransaction(action, name)` (commit-or-rollback), for each change `root.FindById(guid)` and mutate the **live attached instance** at leaf granularity — `obj.SetValue(coreProperty, value)` for CoreProperties (gets `[Range]` coercion free), `IProperty<T>.CurrentValue`/`.Animation`/`.Expression` for engine params, collections via Id-match then `Insert`/`Add`/`RemoveAt`/`Move` (deserialize new items into a fresh instance first; **for keyframes use `animation.KeyFrames.Add(keyframe, out _)`** — the sorted-insertion API on `KeyFrames : HierarchicalList<IKeyFrame>` (`KeyFrames.cs:15`) — and `KeyFrames.Remove`/`RemoveAt` for removal; the convenience `AnimationOperations.AddKeyFrame`/`RemoveKeyFrame` helpers are **UI-only** (`Beutl.Editor.Components`), so the headless core uses the engine `KeyFrames` collection directly); (4) `ExecuteInTransaction` commits once on success or **rolls back on any exception**, so no partial live mutation persists (a bare `Commit` would not undo a mid-reconcile throw, breaking FR-012).

**Rationale**: There is no direct command-construction API — every undoable edit works by **mutating the model while `CoreObjectOperationObserver` listens** (`ValueChanged`/`CollectionChanged`) and emits operations into the current transaction, then the service `Commit`s once (verified in `ElementMoveService`, `SceneSettingsService` — multiple raw writes collapse into one `Commit`). Reusing this path inherits, for free: `[Range]` coercion, animation/expression handling, nested-object + collection identity tracking, lazy observer wiring for inserted children, and operation auto-compaction (`IMergableChangeOperation.TryMerge`).

**Alternatives**: construct `ChangeOperation`s and `Record` them directly — rejected (re-implements PropertyPath construction, registry-vs-engine dispatch, collection bookkeeping). `PopulateFromJsonObject` on the root — rejected (it can `Activator.CreateInstance` new children, destroying `CoreObject.Id` identity and detaching observers — and it bypasses undo). Mutate with observer detached + snapshot a `CustomOperation` — rejected (opaque, no per-field preview/validation/merge). Run `plan` on the live doc then `Rollback()` — rejected (fires real events into UI mirrors/AutoSave/render before rollback); instead **`plan` reconciles on a deep clone** of the root.

**Key facts / risks**: atomic API `HistoryManager.Commit(name?)` / `ExecuteInTransaction(action, name)`; `Record` is suppressed during Undo/Redo. Identity primitive `ICoreObject.FindById(guid)`. Collections: the Scene's elements (CLR `Scene.Children` of type `Elements`, but `[NotAutoSerialized]` — serialized/diffed under the key **`Elements`**), `Element.Objects` (`ICoreList<EngineObject>`), `KeyFrameAnimation.KeyFrames`. **No JSON diff/patch library exists in the repo** — RFC 7396 + the Id-diff are ours. **Validation reporting (FR-007/FR-030/SC-002/SC-009)**: `CoreObject.SetValue` is `void` and coerces-then-records-`Errors`; `IProperty` setters assign coerced values without a return. To report typed *accepted / coerced / rejected*, the reconciler/`plan` must run each property's validator (`CorePropertyMetadata<T>.Validator` → `TryCoerce`/`Validate`) **explicitly** to compute the outcome (and clamped value/range), not infer it post-mutation. Operation introspection for `plan`: `IUpdatePropertyValueOperation` exposes Object/PropertyPath/NewValue/OldValue; collection ops expose Items. Id handling rule: new objects **omit `Id`** (toolkit mints one + inserts); an **unknown supplied `Id`** ⇒ `stale_handle`; an **existing `Id` with a different `$type`** ⇒ `validation_rejected` (no in-place type change — delete + insert). Other risks: keyframe time-sort; single-writer threading; untracked properties (`Tracked=false`, `HierarchicalParentProperty`) skipped by the observer; engine-vs-core property dispatch.

## 4. Schema generation + JSON Merge Patch

**Decision (schema)**: Generate the schema by **runtime introspection of instantiated objects**, covering **both** property systems. Legacy `CoreProperty` (`PropertyRegistry.GetRegistered(type)`) backs base/identity fields (`CoreObject.Id`, `EngineObject.IsEnabled`/`ZIndex`/`TimeRange`); the **modern per-object `IProperty<T>`** surface (e.g. `Brightness.Amount = Property.CreateAnimatable(100f)`) holds the bulk of editable parameters and is populated by `ScanProperties<T>()` in each `EngineObject` ctor, exposed via `EngineObject.Properties`. For each registered implementation type: `Activator.CreateInstance` it, then read each `IProperty`'s `Name`, `ValueType`, `DefaultValue`, `IsAnimatable`, `SupportsExpression`, and `GetAttributes()` (`[Display]` name/desc, `[Range]` min/max, `[NumberStep]` step) — plus base CoreObject `Id` and inherited `EngineObject` CoreProperties. Get the `$type` string via the public `JsonHelper.WriteDiscriminator`/`TryGetDiscriminator` (not the `internal` `TypeFormat`). Enumerate the type universe via `LibraryService.Current` (after running registration in the headless host) or by reflecting `EngineObject`/`CoreObject` subclasses.

**Decision (merge-patch)**: Hand-roll **RFC 7396** (≈30 lines) over `System.Text.Json` `JsonNode` — there is no built-in `JsonNode.Merge` and no suitable in-box `JsonMergePatch`. `Apply(target, patch)`: if `patch` is a `JsonObject`, for each member `null` ⇒ remove key, else recurse; **arrays of `Id`-bearing entities use id-keyed merge** (match by `Id`, leave unmentioned siblings, `$delete` removes one); only scalar / non-identified arrays replace wholesale. **DeepClone** nodes before reparenting (STJ enforces single-parent). Reuse `Beutl.Serialization.JsonDeepClone` where possible. *(This id-keyed array rule is a deliberate extension to strict RFC 7396, required by FR-028 — raw array-replace would drop siblings.)* Ordering: existing member order preserved; new members append unless `$index`/`$after`/`$before`; keyframes ordered by `KeyTime`. `$delete` on a missing `Id` is an idempotent no-op (vs a plain unknown-`Id` member ⇒ `stale_handle`).

**Rationale**: Walking `PropertyRegistry` alone would miss almost all agent-editable fields (the modern `IProperty` system). `IProperty` exposes exactly the schema fields needed and maps 1:1 onto the serialized JSON key shape (`Name` ⇒ property, animations under `"Animations"`, expressions under `"Expressions"`). RFC 7396 (JSON Merge Patch) is the LLM-friendly partial-edit format and is trivial to implement correctly with a focused test.

**Alternatives**: schema from `PropertyRegistry` only — rejected (misses `IProperty`). Pure reflection without instantiation — rejected (can't read inline `DefaultValue`s set in field initializers). Schema from serialized JSON only — rejected (no type/range/default/animatability metadata). A source generator — rejected (can't see runtime plugin types; duplicates runtime metadata). NuGet merge-patch libs (LaDeak/Morcatko) — rejected (Newtonsoft-based or POCO-typed; the repo is all STJ). `JObject.Merge` — rejected (not RFC 7396 — union-merges, doesn't treat null as delete).

**Key facts / risks**: strict RFC 7396 replaces arrays wholesale, but the toolkit applies **id-keyed merge to arrays of `Id`-bearing entities** (elements/objects/keyframes — match by `Id`, leave unmentioned siblings, `$delete` removes); only scalar / non-identified arrays replace wholesale. The merge derives the *desired document* (siblings preserved), and the **Id-based collection diff** then computes minimal operations (§3, contracts §2). Schema gen instantiates each type (needs a public parameterless ctor; guard ctors with side effects). `LibraryService.Current` is empty until registration runs (headless host must trigger it). Some properties carry custom `JsonConverter`s — the schema records the converter/encoding rather than a structural field description.

## 5. Project placement, references, TFM, entrypoint

**Decision**: Add new MIT, single-target `net10.0`, `IsPackable=false` projects under `src/` — two toolkit projects here plus the split-out `Beutl.Extensions.FFmpeg.Core` for export (§2), three new projects in total: **`Beutl.AgentToolkit`** (core library; `ProjectReference`s `Beutl.Editor`, `Beutl.ProjectSystem`, `Beutl.Engine`, `Beutl.Extensibility` — `Beutl.Core` arrives transitively) and **`Beutl.AgentToolkit.Mcp`** (console `Exe`; references only the core lib + `ModelContextProtocol` + `Microsoft.Extensions.Hosting`). The core does **not** reference `Beutl.Extensions.FFmpeg` directly (Avalonia-coupled); for the built-in encoder it references the new MIT **non-UI** assembly `Beutl.Extensions.FFmpeg.Core` (the split-out encode core), and loads installed encoders via a public headless registration service — all behind `Beutl.Extensibility.EncodingController`. `Beutl.FFmpegIpc` is reached transitively through that non-UI encoder assembly, not as a direct core edge. Register all three projects in `Beutl.slnx`; add the packages (`ModelContextProtocol`, `ModelContextProtocol.AspNetCore`, `Microsoft.Extensions.Hosting`) to `Directory.Packages.props` (central package management — neither is present yet). Add `tests/Beutl.AgentToolkit.Tests` referencing the **library** (not the exe). The MCP entrypoint is the stdio host from §1.

**Rationale**: Splitting core-lib from server-exe keeps the logic unit-testable without a transport and reusable by other front-ends — mirroring the repo's `Beutl.Editor` / `Beutl.Editor.Components` split. `net10.0` single-target is correct: the whole dependency closure is single-target `net10.0`, and the toolkit uses no Windows-only API. `Beutl.Editor` already transitively pulls Engine/Extensibility/ProjectSystem/Core, so the core has **no genuinely new compile-time edge** — video export is reached via the `Beutl.Extensibility` encoder abstraction with the concrete encoder loaded at runtime, avoiding a direct reference to the Avalonia-coupled `Beutl.Extensions.FFmpeg` or to `Beutl.FFmpegIpc`.

**Alternatives**: a single combined exe — rejected (core can't be cleanly unit-tested in an entrypoint assembly, no reuse). Dual-target like the worker/UI — rejected (no dependency dual-targets; no Windows-only API). Put it inside `Beutl.Editor` — rejected (adds MCP/Hosting to a core editing assembly; orthogonality). Name them `Beutl.Mcp`/`Beutl.Mcp.Core` — acceptable but capability-naming (`Beutl.AgentToolkit`) leaves room for a non-MCP front-end.

**Key facts / risks**: `Directory.Build.props` sets `net10.0`/`preview`/`Nullable enable` for all projects. The boundary hook denies any `ProjectReference` to `Beutl.FFmpegWorker` (toolkit is compliant by construction). The MCP exe is the **first MIT console `Exe` in `src/`** — confirm CI build/publish (Nuke) and release packaging don't assume every `src/` exe is the GPL worker. `Microsoft.Extensions.Hosting` is not yet in `Directory.Packages.props` (pins `Microsoft.Extensions.*` at 10.0.9) — add it and the SDK, watch for a version-conflict warning.

## 6. Workspace sandbox + GPU-free test strategy

**Decision (sandbox)**: Route every write through one chokepoint `IWorkspaceGuard.ResolveForWrite(requestedPath) -> canonicalAbsolutePath`; reads bypass it entirely. Canonicalize in two stages (the target usually does not exist yet): (1) `Path.GetFullPath` against the configured root (lexical `.`/`..`, no FS access); (2) walk to the deepest **existing** ancestor and resolve symlinks/junctions on it with `Directory/File.ResolveLinkTarget(path, returnFinalTarget: true)`, then re-append the non-existent remainder. Containment: append `Path.DirectorySeparatorChar` to the canonical root and test `StartsWith` with `OrdinalIgnoreCase` on Windows/macOS, `Ordinal` on Linux. Reject failures — including an in-root symlink that resolves outside — with a typed `WorkspaceBoundaryException`.

**Decision (tests)**: The substantive logic is Avalonia-free and GPU-free → unit-test it in `tests/Beutl.AgentToolkit.Tests`: schema; **merge-patch incl. id-keyed array edge cases (`$delete`; `$delete` on missing `Id` ⇒ no-op; omitted-`Id` insert; unknown-`Id` ⇒ `stale_handle`; same-`Id`/different-`$type` ⇒ `validation_rejected`; ordering `$index`/`$after`/`$before` ⇒ `move-child`; multiple directives ⇒ `validation_rejected`; bad sibling ⇒ `stale_handle`; keyframes by time)**; **reconcile→operations incl. a mid-reconcile `ExecuteInTransaction` rollback case**; capability discovery; the workspace guard. Render/export tests that need a GPU self-skip via the existing `VulkanTestEnvironment`/`GpuTestEnvironment` idiom (`Assert.Ignore` when `GraphicsContextFactory.GetOrCreateShared()` is null). Export orchestration is testable worker-free over a real `System.IO.Pipes` pair with a fake host (as `tests/Beutl.FFmpegIpc.Tests` already does).

**Rationale**: Path containment cannot be a string prefix — `Path.GetFullPath` never resolves symlinks, so an in-root link pointing out would pass a naive check; hence `ResolveLinkTarget` on the deepest existing ancestor + per-OS case sensitivity + trailing separator. A single typed-exception chokepoint keeps the read-anywhere / write-in-root asymmetry auditable and lets the apply layer treat a boundary violation like any failed operation and roll back via `HistoryManager`.

**Alternatives**: `GetFullPath` + `StartsWith` only — rejected (symlink escape). `GetRelativePath` + reject leading `..` — rejected (still lexical; useful only as a cheap pre-filter). OS sandbox (chroot/container/seccomp) — rejected (out of scope for an in-process embeddable cross-platform library). All tests in `tests/Beutl.UnitTests` — allowed but a dedicated project is cleaner. Export tests through `HeadlessUITests` — rejected (that suite references the app shell; the toolkit's export is programmatic).

**Key facts / risks**: `ResolveLinkTarget` (NET 6+) follows up to 40 links (Unix) / 63 (Windows); known edge cases on relative links over UNC and non-existent components (resolve only existing components). Windows symlink creation needs Developer Mode/admin → escape-symlink test fixtures self-skip on `UnauthorizedAccessException`. macOS case-sensitivity is a heuristic (`OrdinalIgnoreCase` is the stricter, safe default). TOCTOU window between canonicalize and open — re-validate or open with no-follow where possible. A true end-to-end export test needs both a GPU and the FFmpeg worker binary, so it will usually skip on CI — keep the reconciliation/orchestration logic factored out and unit-tested separately.

## 7. Real-time UI reflection (in-app hosting)

**Decision**: Deliver real-time reflection by hosting the toolkit core **in-process inside the running Beutl editor**, bound to the active `EditViewModel`'s existing live session (its `Scene` + `HistoryManager` + `CoreObjectOperationObserver`), reached by an external agent over a **loopback HTTP/SSE** MCP endpoint (`ModelContextProtocol.AspNetCore`). The headless stdio server stays for the no-GUI case; both share the `Beutl.AgentToolkit` core via the *session-source* seam (file-opened vs live). Do **not** pursue file-watch reload or a cross-process scene-sharing IPC.

**Rationale**: The live model→UI path already exists — a model mutation raises `PropertyChanged`/collection events (`CoreObject.cs:252-270`) that Avalonia reactive bindings + the `Renderer`/`Composer` reactive rebuilds (`EditViewModel.cs:86-101`) turn into a repaint; the timeline binds the same way. So if the agent mutates the **same live `Scene`** the GUI holds, the UI updates live "for free". Hosting in-process is the cheapest way to share that instance: there is exactly **one** `HistoryManager` (the editor's), so the agent is just an additional command source — no two-writer history corruption, and **no IPC layer to build** (Beutl has none — see alternatives). Edits also become normal, human-undoable history entries.

**Alternatives**: file-watch reload (a new `FileSystemWatcher` in `ProjectService` → `CoreSerializer.RestoreFromUri` → swap the `Scene`) — rejected as the *primary* path: it gives only coarse, on-save "eventual" reflection and forces a dirty-state conflict resolution when the GUI has unsaved edits (which version wins). Cross-process shared `Scene` / a new IPC channel forwarding edits to the GUI — rejected: Beutl has **zero IPC infrastructure** into the running app (no named pipes/sockets/MMF/HTTP, no single-instance, no file-association handler — confirmed), and two `HistoryManager`s over one `Scene` would corrupt undo/redo. In-app hosting reduces this to one writer in one process.

**Key facts / risks**: No project-file `FileSystemWatcher`/reload exists (`ProjectService.OpenProject` deserializes once). The GUI owns its sole in-memory `Scene` per `EditViewModel`; `AutoSaveService` is per-`EditViewModel`, dirty state ≈ `HistoryManager.HasPendingOperations`. Risks: the in-app endpoint is a self-hosted Kestrel loopback server inside a desktop app (bind `127.0.0.1`, per-session token, clean lifecycle); mutations must be marshaled onto the editor's writer/UI thread; the live session is valid only while the scene is open (typed "no active editor session" otherwise); this is *live observation of the shared model*, not GUI/pixel automation (which stays out of scope).
