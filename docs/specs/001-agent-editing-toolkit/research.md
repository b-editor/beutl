# Phase 0 Research: Agent Editing Toolkit

Six unknowns from the plan's Technical Context were investigated against the current codebase and the .NET ecosystem. Each is recorded as Decision / Rationale / Alternatives, with the concrete facts that ground the design. No `NEEDS CLARIFICATION` remained.

---

## 1. MCP hosting in .NET

**Decision**: Build the MCP server as a .NET console exe using the official **`ModelContextProtocol`** NuGet package (1.4.0, MIT, maintained by Anthropic + Microsoft) with the Generic-Host model and **stdio** transport: `Host.CreateApplicationBuilder(args)` → `builder.Services.AddMcpServer().WithStdioServerTransport().WithToolsFromAssembly()` → `RunAsync()`. Tools are `[McpServerToolType]` classes with `[McpServerTool]` methods; `[Description]` on methods/parameters documents the tool and its input schema. Beutl services are injected as method parameters from the host DI container. `IMcpServer`, `CancellationToken`, `IProgress<>` are framework-injected. For `plan`/`apply`, return typed records so the SDK emits an output schema + structured content; for simple tools, return a serialized JSON string.

**Rationale**: The attribute + Generic-Host style is the SDK's documented happy path and maps onto Beutl's constructor-injection world with no manual JSON-RPC plumbing. Stdio is exactly what Claude Code's `.mcp.json` launcher spawns — no port, no auth, no ASP.NET Core. The package is MIT and pulls only `Microsoft.Extensions.*`; nothing reaches the GPL worker.

**Alternatives**: `ModelContextProtocol.AspNetCore` (HTTP/SSE) — rejected, adds a Kestrel listener/auth for a local spawned exe; keep for a future shared-service mode. `ModelContextProtocol.Core` only — rejected, loses `WithToolsFromAssembly`/DI sugar. Hand-rolled JSON-RPC — rejected, re-implements framing/capability/schema the SDK maintains. Third-party alpha SDKs — rejected (unofficial).

**Key facts / gotchas**: latest stable `ModelContextProtocol` 1.4.0; also needs `Microsoft.Extensions.Hosting`. **CRITICAL**: route all logging to **STDERR** (`builder.Logging.AddConsole(o => o.LogToStandardErrorThreshold = LogLevel.Trace)`) — STDOUT is the JSON-RPC channel; any stray STDOUT write corrupts the protocol. `.mcp.json` stdio entry: `{ "command": "dotnet", "args": ["run","--project","<csproj>"], "env": { "BEUTL_WORKSPACE": "<root>" } }` (or the published exe path). Open risks: structured-output ergonomics (typed record vs serialized string) and net8/net9-SDK-TFM consumed from net10.0 — confirm at first build; pin the version; dispose Beutl services cleanly on stdin EOF.

## 2. Headless rendering & video export, and the GPL boundary

**Decision**: Both a still-frame PNG and a full audio+video export are feasible headlessly on the MIT side, with no Avalonia/window/UI thread. **Still**: `SceneRenderer(scene)` → `frame.Compositor.EvaluateGraphics(time)` → `Renderer.Render(frame)` → `Renderer.Snapshot()` (`Beutl.Media.Bitmap`) → `Bitmap.Save(path, EncodedImageFormat.Png)`. **Video**: replicate the orchestration in `OutputViewModel.StartEncode` inside a small headless host — build `SceneRenderer` + `FrameProviderImpl` (video) + `SceneComposer` + `SampleProviderImpl` (audio) and call `EncodingController.Encode(frameProvider, sampleProvider, ct)`. Relocate `FrameProviderImpl`/`SampleProviderImpl` (currently in the Avalonia app `src/Beutl/Models`, but Avalonia-free) into `Beutl.Editor`.

**Rationale**: Rendering already runs on `RenderThread.Dispatcher` (a `Dispatcher.Spawn`'d background thread, no UI thread). The encode loop is plain async engine code; only progress `ReactiveProperty`/`NotificationService` are GUI-bound and a headless host omits them. The MIT→GPL reach is already correct: `FFmpegEncodingControllerProxy` spawns `Beutl.FFmpegWorker` as a **separate process** and talks to it only over `Beutl.FFmpegIpc` (named pipe + shared memory) — no `ProjectReference`.

**Alternatives**: reference `src/Beutl` to reuse `OutputViewModel` — rejected (the app project is referenced only by `HeadlessUITests` by policy; `OutputViewModel` is `EditViewModel`-bound). Drive the live OutputTab UI — rejected (out of scope; the encode path needs no UI). Call FFmpeg/the worker directly — rejected (crosses the GPL boundary).

**Key facts / risks**: `RenderTarget.Create` uses the GPU (Vulkan; macOS = MoltenVK) when available and **falls back to a CPU SkiaSharp surface** when no device/Dispatcher (`GraphicsContextFactory.GetOrCreateShared()` returns null on failure). CPU fallback covers vector/text/bitmap stills; **GLSL/SKSL FilterEffects, particles, and 3D need a real GPU** — surface a typed "rendering unavailable" error and gate those tests. FFmpeg **native libraries are a runtime prerequisite** (worker throws `FFmpegLibrariesNotFoundException`/exit 2) — preflight + typed error. Rendering must be marshaled onto `RenderThread.Dispatcher` (naive off-dispatcher calls hit `VerifyAccess`). Export supersampling/working-scale (`WorkingScaleCeiling.Export()` = +∞) is bounded by `ClampWorkingScaleToBufferBudget` — reuse the existing clamp.

## 3. Reconciliation: desired-state / merge-patch → undoable operations

**Decision**: Reconcile on Beutl's **observer-driven recording pipeline**, not by hand-constructing `ChangeOperation`s and not by `PopulateFromJsonObject` on the live tree. Per session, stand up the trio the editor uses (`EditViewModel.cs:117-127`): `var gen = new OperationSequenceGenerator(); var history = new HistoryManager(root, gen); var observer = new CoreObjectOperationObserver(null, root, gen); history.Subscribe(observer);`. To reconcile: (1) serialize the live root via `CoreSerializer.SerializeToJsonObject`; for a merge-patch request, apply RFC 7396 to get the desired document; (2) structurally diff current-vs-desired into a leaf change list keyed by `CoreObject.Id` + property name; (3) inside ONE uncommitted transaction, for each change `root.FindById(guid)` and mutate the **live attached instance** at leaf granularity — `obj.SetValue(coreProperty, value)` for CoreProperties (gets `[Range]` coercion free), `IProperty<T>.CurrentValue`/`.Animation`/`.Expression` for engine params, collections via Id-match then `Insert`/`Add`/`RemoveAt`/`Move` (deserialize new items into a fresh instance first; prefer `KeyFrameAnimation.AddKeyFrame`/`RemoveKeyFrame` to keep the time-sort); (4) `history.Commit(name)` exactly once.

**Rationale**: There is no direct command-construction API — every undoable edit works by **mutating the model while `CoreObjectOperationObserver` listens** (`ValueChanged`/`CollectionChanged`) and emits operations into the current transaction, then the service `Commit`s once (verified in `ElementMoveService`, `SceneSettingsService` — multiple raw writes collapse into one `Commit`). Reusing this path inherits, for free: `[Range]` coercion, animation/expression handling, nested-object + collection identity tracking, lazy observer wiring for inserted children, and operation auto-compaction (`IMergableChangeOperation.TryMerge`).

**Alternatives**: construct `ChangeOperation`s and `Record` them directly — rejected (re-implements PropertyPath construction, registry-vs-engine dispatch, collection bookkeeping). `PopulateFromJsonObject` on the root — rejected (it can `Activator.CreateInstance` new children, destroying `CoreObject.Id` identity and detaching observers — and it bypasses undo). Mutate with observer detached + snapshot a `CustomOperation` — rejected (opaque, no per-field preview/validation/merge). Run `plan` on the live doc then `Rollback()` — rejected (fires real events into UI mirrors/AutoSave/render before rollback); instead **`plan` reconciles on a deep clone** of the root.

**Key facts / risks**: atomic API `HistoryManager.Commit(name?)` / `ExecuteInTransaction(action, name)`; `Record` is suppressed during Undo/Redo. Identity primitive `ICoreObject.FindById(guid)`. Collections: `Scene.Children` (Elements), `Element.Objects` (`ICoreList<EngineObject>`), `KeyFrameAnimation.KeyFrames`. **No JSON diff/patch library exists in the repo** — RFC 7396 + the Id-diff are ours. Operation introspection for `plan`: `IUpdatePropertyValueOperation` exposes Object/PropertyPath/NewValue/OldValue; collection ops expose Items. Risks: new/duplicate/unknown `Id`s; keyframe time-sort; single-writer threading; untracked properties (`Tracked=false`, `HierarchicalParentProperty`) skipped by the observer; engine-vs-core property dispatch.

## 4. Schema generation + JSON Merge Patch

**Decision (schema)**: Generate the schema by **runtime introspection of instantiated objects**, covering **both** property systems. Legacy `CoreProperty` (`PropertyRegistry.GetRegistered(type)`) backs base/identity fields (`CoreObject.Id`, `EngineObject.IsEnabled`/`ZIndex`/`TimeRange`); the **modern per-object `IProperty<T>`** surface (e.g. `Brightness.Amount = Property.CreateAnimatable(100f)`) holds the bulk of editable parameters and is populated by `ScanProperties<T>()` in each `EngineObject` ctor, exposed via `EngineObject.Properties`. For each registered implementation type: `Activator.CreateInstance` it, then read each `IProperty`'s `Name`, `ValueType`, `DefaultValue`, `IsAnimatable`, `SupportsExpression`, and `GetAttributes()` (`[Display]` name/desc, `[Range]` min/max, `[NumberStep]` step) — plus base CoreObject `Id` and inherited `EngineObject` CoreProperties. Get the `$type` string via the public `JsonHelper.WriteDiscriminator`/`TryGetDiscriminator` (not the `internal` `TypeFormat`). Enumerate the type universe via `LibraryService.Current` (after running registration in the headless host) or by reflecting `EngineObject`/`CoreObject` subclasses.

**Decision (merge-patch)**: Hand-roll **RFC 7396** (≈30 lines) over `System.Text.Json` `JsonNode` — there is no built-in `JsonNode.Merge` and no suitable in-box `JsonMergePatch`. `Apply(target, patch)`: if `patch` is a `JsonObject`, for each member `null` ⇒ remove key, else recurse; arrays/scalars replace wholesale. **DeepClone** nodes before reparenting (STJ enforces single-parent). Reuse `Beutl.Serialization.JsonDeepClone` where possible.

**Rationale**: Walking `PropertyRegistry` alone would miss almost all agent-editable fields (the modern `IProperty` system). `IProperty` exposes exactly the schema fields needed and maps 1:1 onto the serialized JSON key shape (`Name` ⇒ property, animations under `"Animations"`, expressions under `"Expressions"`). RFC 7396 (which obsoletes 7386, same algorithm) is the LLM-friendly partial-edit format and is trivial to implement correctly with a focused test.

**Alternatives**: schema from `PropertyRegistry` only — rejected (misses `IProperty`). Pure reflection without instantiation — rejected (can't read inline `DefaultValue`s set in field initializers). Schema from serialized JSON only — rejected (no type/range/default/animatability metadata). A source generator — rejected (can't see runtime plugin types; duplicates runtime metadata). NuGet merge-patch libs (LaDeak/Morcatko) — rejected (Newtonsoft-based or POCO-typed; the repo is all STJ). `JObject.Merge` — rejected (not RFC 7396 — union-merges, doesn't treat null as delete).

**Key facts / risks**: **RFC 7396 replaces arrays wholesale (no element identity)** — so a merge-patch is applied to the JSON to derive the *desired document*, and the **Id-based collection diff is a separate step** in the reconciler (§3). Schema gen instantiates each type (needs a public parameterless ctor; guard ctors with side effects). `LibraryService.Current` is empty until registration runs (headless host must trigger it). Some properties carry custom `JsonConverter`s — the schema records the converter/encoding rather than a structural field description.

## 5. Project placement, references, TFM, entrypoint

**Decision**: Add two new MIT, single-target `net10.0`, `IsPackable=false` projects under `src/`: **`Beutl.AgentToolkit`** (core library; `ProjectReference`s `Beutl.Editor`, `Beutl.ProjectSystem`, `Beutl.Engine`, `Beutl.Extensibility`, `Beutl.FFmpegIpc`) and **`Beutl.AgentToolkit.Mcp`** (console `Exe`; references only the core lib + `ModelContextProtocol` + `Microsoft.Extensions.Hosting`). Register both in `Beutl.slnx`; add the two packages to `Directory.Packages.props` (central package management — neither is present yet). Add `tests/Beutl.AgentToolkit.Tests` referencing the **library** (not the exe). The MCP entrypoint is the stdio host from §1.

**Rationale**: Splitting core-lib from server-exe keeps the logic unit-testable without a transport and reusable by other front-ends — mirroring the repo's `Beutl.Editor` / `Beutl.Editor.Components` split. `net10.0` single-target is correct: the whole dependency closure is single-target `net10.0`, and the toolkit uses no Windows-only API. `Beutl.Editor` already transitively pulls Engine/Extensibility/ProjectSystem/Core; `Beutl.FFmpegIpc` is the only genuinely new edge (listing the others explicitly documents intent).

**Alternatives**: a single combined exe — rejected (core can't be cleanly unit-tested in an entrypoint assembly, no reuse). Dual-target like the worker/UI — rejected (no dependency dual-targets; no Windows-only API). Put it inside `Beutl.Editor` — rejected (adds MCP/Hosting to a core editing assembly; orthogonality). Name them `Beutl.Mcp`/`Beutl.Mcp.Core` — acceptable but capability-naming (`Beutl.AgentToolkit`) leaves room for a non-MCP front-end.

**Key facts / risks**: `Directory.Build.props` sets `net10.0`/`preview`/`Nullable enable` for all projects. The boundary hook denies any `ProjectReference` to `Beutl.FFmpegWorker` (toolkit is compliant by construction). The MCP exe is the **first MIT console `Exe` in `src/`** — confirm CI build/publish (Nuke) and release packaging don't assume every `src/` exe is the GPL worker. `Microsoft.Extensions.Hosting` is not yet in `Directory.Packages.props` (pins `Microsoft.Extensions.*` at 10.0.9) — add it and the SDK, watch for a version-conflict warning.

## 6. Workspace sandbox + GPU-free test strategy

**Decision (sandbox)**: Route every write through one chokepoint `IWorkspaceGuard.ResolveForWrite(requestedPath) -> canonicalAbsolutePath`; reads bypass it entirely. Canonicalize in two stages (the target usually does not exist yet): (1) `Path.GetFullPath` against the configured root (lexical `.`/`..`, no FS access); (2) walk to the deepest **existing** ancestor and resolve symlinks/junctions on it with `Directory/File.ResolveLinkTarget(path, returnFinalTarget: true)`, then re-append the non-existent remainder. Containment: append `Path.DirectorySeparatorChar` to the canonical root and test `StartsWith` with `OrdinalIgnoreCase` on Windows/macOS, `Ordinal` on Linux. Reject failures — including an in-root symlink that resolves outside — with a typed `WorkspaceBoundaryException`.

**Decision (tests)**: The substantive logic (schema, merge-patch, reconcile→operations, capability discovery, guard) is Avalonia-free and GPU-free → unit-test it in `tests/Beutl.AgentToolkit.Tests`. Render/export tests that need a GPU self-skip via the existing `VulkanTestEnvironment`/`GpuTestEnvironment` idiom (`Assert.Ignore` when `GraphicsContextFactory.GetOrCreateShared()` is null). Export orchestration is testable worker-free over a real `System.IO.Pipes` pair with a fake host (as `tests/Beutl.FFmpegIpc.Tests` already does).

**Rationale**: Path containment cannot be a string prefix — `Path.GetFullPath` never resolves symlinks, so an in-root link pointing out would pass a naive check; hence `ResolveLinkTarget` on the deepest existing ancestor + per-OS case sensitivity + trailing separator. A single typed-exception chokepoint keeps the read-anywhere / write-in-root asymmetry auditable and lets the apply layer treat a boundary violation like any failed operation and roll back via `HistoryManager`.

**Alternatives**: `GetFullPath` + `StartsWith` only — rejected (symlink escape). `GetRelativePath` + reject leading `..` — rejected (still lexical; useful only as a cheap pre-filter). OS sandbox (chroot/container/seccomp) — rejected (out of scope for an in-process embeddable cross-platform library). All tests in `tests/Beutl.UnitTests` — allowed but a dedicated project is cleaner. Export tests through `HeadlessUITests` — rejected (that suite references the app shell; the toolkit's export is programmatic).

**Key facts / risks**: `ResolveLinkTarget` (NET 6+) follows up to 40 links (Unix) / 63 (Windows); known edge cases on relative links over UNC and non-existent components (resolve only existing components). Windows symlink creation needs Developer Mode/admin → escape-symlink test fixtures self-skip on `UnauthorizedAccessException`. macOS case-sensitivity is a heuristic (`OrdinalIgnoreCase` is the stricter, safe default). TOCTOU window between canonicalize and open — re-validate or open with no-follow where possible. A true end-to-end export test needs both a GPU and the FFmpeg worker binary, so it will usually skip on CI — keep the reconciliation/orchestration logic factored out and unit-tested separately.

## 7. Real-time UI reflection (in-app hosting)

**Decision**: Deliver real-time reflection by hosting the toolkit core **in-process inside the running Beutl editor**, bound to the active `EditViewModel`'s existing live session (its `Scene` + `HistoryManager` + `CoreObjectOperationObserver`), reached by an external agent over a **loopback HTTP/SSE** MCP endpoint (`ModelContextProtocol.AspNetCore`). The headless stdio server stays for the no-GUI case; both share the `Beutl.AgentToolkit` core via the *session-source* seam (file-opened vs live). Do **not** pursue file-watch reload or a cross-process scene-sharing IPC.

**Rationale**: The live model→UI path already exists — a model mutation raises `PropertyChanged`/collection events (`CoreObject.cs:252-270`) that Avalonia reactive bindings + the `Renderer`/`Composer` reactive rebuilds (`EditViewModel.cs:86-101`) turn into a repaint; the timeline binds the same way. So if the agent mutates the **same live `Scene`** the GUI holds, the UI updates live "for free". Hosting in-process is the cheapest way to share that instance: there is exactly **one** `HistoryManager` (the editor's), so the agent is just an additional command source — no two-writer history corruption, and **no IPC layer to build** (Beutl has none — see alternatives). Edits also become normal, human-undoable history entries.

**Alternatives**: file-watch reload (a new `FileSystemWatcher` in `ProjectService` → `CoreSerializer.RestoreFromUri` → swap the `Scene`) — rejected as the *primary* path: it gives only coarse, on-save "eventual" reflection and forces a dirty-state conflict resolution when the GUI has unsaved edits (which version wins). Cross-process shared `Scene` / a new IPC channel forwarding edits to the GUI — rejected: Beutl has **zero IPC infrastructure** into the running app (no named pipes/sockets/MMF/HTTP, no single-instance, no file-association handler — confirmed), and two `HistoryManager`s over one `Scene` would corrupt undo/redo. In-app hosting reduces this to one writer in one process.

**Key facts / risks**: No project-file `FileSystemWatcher`/reload exists (`ProjectService.OpenProject` deserializes once). The GUI owns its sole in-memory `Scene` per `EditViewModel`; `AutoSaveService` is per-`EditViewModel`, dirty state ≈ `HistoryManager.HasPendingOperations`. Risks: the in-app endpoint is a self-hosted Kestrel loopback server inside a desktop app (bind `127.0.0.1`, per-session token, clean lifecycle); mutations must be marshaled onto the editor's writer/UI thread; the live session is valid only while the scene is open (typed "no active editor session" otherwise); this is *live observation of the shared model*, not GUI/pixel automation (which stays out of scope).
