# Contract: Graph Compilation & Execution

**Feature**: `004-gpu-pass-fusion` | Binding on: the compiler (`EffectGraph` → `CompiledPlan`) and `PlanExecutor`

## C1. Fusion rules (normative)

1. A **fusion group** is a maximal run of adjacent nodes on one edge-chain where every node is coordinate-invariant (`ShaderNode` snippet with `IsCoordinateInvariant`, or `ColorFilterNode`).
2. One fusion group compiles to **one `FusedShaderPass`**: one draw into one target, built by shader composition (input image shader → alternating `WithColorFilter` wraps and `SKRuntimeEffect` child-shader wraps, in node order). Node order is strictly preserved.
3. Adjacent SKSL snippets within a group SHOULD be merged into one generated program (uniforms prefixed `fe{N}_`, chained `apply` calls); merging is an optimization and MUST NOT change ordering or results beyond floating-point tolerance.
4. Fusion groups never cross: a non-invariant node, a `SplitNode`/`CompositeNode`, a backend transition, or a group that would exceed the composition budget (max 16 shader stages or backend uniform limits — the compiler splits into consecutive fused passes, FR-005).
5. A fusion group at the head of a chain samples the group's input exactly once per output pixel; a group's pass ROI equals the intersection of the requested output ROI with the group bounds (identity transforms).
   > **Status note (2026-07):** only the **parent's requested-ROI intersection at the graph output** is production-latent — `PlanFilterEffectRenderNode.Process` passes `Rect.Invalid` to `ResolveResources`, so the *last* pass is seeded from its own `OutputBounds` rather than an externally-requested crop (exercised by the `ResolveResources` unit tests, not by production renders). Plumbing a real production ROI from the parent is a maintainer-deferred follow-up. This does **not** make the internal backward walk latent: see the C3.1 note.

## C2. Skia filter grouping

Adjacent `SkiaFilterNode`s compose into a single `SkiaFilterPass` (one filtered draw), reproducing the pre-redesign `SKImageFilterBuilder` accumulation (now removed) as an explicit plan pass. A `ColorFilterNode` adjacent to a Skia filter run MAY fold into the same paint (Skia applies color filters in the same draw) when node order allows; otherwise it joins the neighboring fusion group.

## C3. Resource planning

1. The compiled `ResourcePlan` declares the **structural shape** of every intermediate (`Id`, format, `[FirstUse, LastUse]`): each pass's outputs (one per current operation, read by the next pass) **and** its statically bounded pass-scoped scratch — the materialized input a geometry/compute/split pass bakes, the **source-halo bake buffer** of a non-invariant whole-source fused pass (acquired only when a downstream deflate narrows the pass output below its backward-claimed input rect `op.Bounds ∪ outBounds`; the source then bakes into that buffer and the `src` shader's local matrix re-registers image coordinates to the output's device space, so crop-after-shift equals shift-then-crop — the contained common case bakes in place with zero extra buffer, keeping the frozen references byte-identical), and the C3.3 ping-pong pair plus depth attachment for compute. Peak-live = interval overlap; the executor MUST NOT exceed it, and it measures this at runtime — the pool tracks concurrently live leases and the executor debug-asserts the measured per-execution peak against the declared bound (the FR-007 gate compares measured peaks of a 10-effect vs a 3-effect chain). Concrete device sizes and per-pass ROIs are computed **every frame** by the resource-resolution pass (pure `Rect` math over the freshly described bounds) — parameter-driven bounds (animated blur sigma, stroke pen, split geometry) MUST flow through without recompilation.
   > **Status note (2026-07):** the internal backward-ROI walk is **always active in production**, not latent. Even with a `Rect.Invalid` request, `ResolveLastRoi` seeds the last pass from its own `OutputBounds`, and the walk then propagates concrete — possibly **deflating** — ROIs to every upstream intermediate pass. So an intermediate pass CAN be ROI-cropped by a downstream deflating pass (a fixed `Clipping`, `DropShadowOnly`) in real renders: an author of a non-invariant pass whose uniforms are anchored in the full-frame device space (a tile grid, a pivot, a cross-sampled child) MUST declare a `RenderTime` bounds contract (or author ROI-relative) so it bakes full-frame regardless of the crop — otherwise the baked sub-rect and the absolute uniforms disagree. Only the **parent's** requested-ROI intersection at the graph *output* (C1.5) is latent; that is a separate, maintainer-deferred plumbing task.
2. Device sizes obey the 003 budget with **legacy carry parity** (FR-012): the working scale is resolved once at the effect boundary (`ResolveWorkingScale`); resource resolution then re-clamps per buffer (`ClampWorkingScaleToBufferBudget`, 16 384 px/axis) in schedule order with a **monotonically non-increasing `w` carried along each chain** — a clamped materializing pass propagates its reduced `w` to downstream passes (the pre-redesign `FilterEffectActivator.Flush` mutated the activator's `WorkingScale`), while intra-pass allocations (ping-pong, geometry-session targets) re-clamp locally without affecting the carry (as the pre-redesign `CustomFilterEffectContext.CreateTarget` did). Same functions, same resulting densities as the pre-redesign pipeline. A **nested-graph branch** (research D8) is a materializing single-op path like any other: the executor describes, resolves, and recursively executes each branch at the **carried** working scale of the operation feeding it (`min(boundary w, the op's EffectiveScale)`; unbounded vector ops take the boundary scale) — never the raw outer working scale, which would re-raise a density an upstream clamped op already reduced.
3. **Uniform values AND density-dependent child shaders are applied at execution, against the pass's resolved density.** When the executor builds a fused pass's shader (`PlanExecutor.BuildRuntimeRun`), it hands the same `PassUniformContext` — carrying the pass's actual `WorkingScale` and output buffer size (`TargetWidth`/`TargetHeight`), the density resolved for THIS pass by point 2 above, after any re-clamp/carry — to both every `UniformBinding` and every `ChildBinding`. A uniform binding whose device-space value depends on that density or size (`DensityScaledFloat2`, `Deferred`) computes it here; a `ChildBinding.Deferred` child (a cross-sampled displacement map) has its shader *product* built here from the same context, so a pass executing below its describe-time working scale gets a coherent offset/pivot/resolution AND a coherently-scaled map lookup (authoring contract A4). The deferred child product is executor-owned: it is tracked on the pass's disposable list and freed after the pass's draw, alongside the built runtime shader (an eager child stays graph-/caller-owned and is left alone). Structural fields (uniform/child names, shader source) are untouched — this is a per-frame parameter resolution, never a recompile.
4. Ping-pong iteration uses exactly 2 declared color targets (+1 depth if required) regardless of iteration count.
5. Empty resolved ROI ⇒ the executor skips the pass at runtime (spec edge case "zero-size targets"); the plan itself is unchanged. The two skip causes differ: an empty resolved **output** (a shrinking pass, e.g. a fully-closed clip) produces **no** downstream operation — the pass's input is released, matching the legacy `Clipping.Apply` target removal — while an empty **input** to an identity pass passes the already-empty operation through.
   - **Render-time output tightening.** A `RenderTime` geometry pass is sized to its full input bounds (the resolver cannot resolve its output forward), so a pass that narrows its content only at render time — `Clipping.AutoClip`, whose margins are detected by sampling the input — calls `GeometrySession.SetOutputBounds(Rect)` inside the callback to emit a **sub-rect** of the allocated buffer. The executor copies that sub-rect into a tighter pooled output target, so the downstream operation's bounds and hit-test region match the content (the legacy `Clipping.Apply` sized its output target to the detected clip rect). Counters: the extra pooled acquire counts one `TargetAllocations`/`PoolAcquires` (C8); the geometry pass still counts one `GpuPass` (the blit is part of emitting the pass output, not a new pass). Peak-live: the executor acquires the tighter target and blits **while the full output is still leased**, then releases the full output — this reuses the concurrency slot the materialized input held during render (input-scratch is already released by then), so the pass's concurrent-lease peak is unchanged and the ResourcePlan `GeometryPass` declaration (§C3.1) is not widened. The requested rect MUST be a shrink (contained in the pass bounds); a grow throws, and a degenerate (empty) request drops the output like `DiscardOutput`, which supersedes a shrink regardless of order. **Downstream propagation:** a later linear non-invariant pass detects that its actual input op is narrower than the resolver's expected input (the previous pass's resolved output ROI) and re-derives its bake window as `resolvedRoi ∩ ForwardBounds(op.Bounds)` — shrink-only, the resolve-time ROI stays the upper bound — so the tightening survives through bounds-inflating successors (an AutoClip → DropShadow publishes bounds derived from the tightened rect, not the frame-start resolution). When the op matches the expectation the resolved ROI is kept verbatim: a forward map is authored against the pass's semantic input frame (a fixed `Clipping` deflates whatever rect it receives), so applying it to a merely ROI-narrowed op would double-apply the crop.
6. **Dynamic outputs**: a pass declared `IsDynamicOutputs` (execution-time-resolved output count, e.g. contour-based part splitting, a nested-graph branch, or an `CustomRenderNodePass`) has no static output decls; the executor allocates its outputs from the pool at execution time, releases them within the frame (leak-asserted), and counts them. The FR-007 static peak-live bound covers declared intermediates only.
   - **Custom-render-node pass (`CustomRenderNodePass`)**: the declarative home for an effect whose execution lives in a custom `FilterEffectRenderNode` (the canonical case is `NodeGraphFilterEffect`, which evaluates a node graph and never describes through the primitive vocabulary). Such an effect derives from the `CustomRenderNodeFilterEffect` base, whose sealed `Describe` calls `EffectGraphBuilder.CustomRenderNode(FilterEffect.Resource child)` to append it, so such an effect is describable **everywhere** — a `FilterEffectGroup`, a `DelayAnimationEffect` branch, any container that walks its children through `Describe` — with no per-container special-casing (before this, `NodeGraphFilterEffect.Describe` threw `NotSupportedException`, crashing any group/delay that held one). Semantics: **render-time resolved** (its output bounds are unknown until the child node runs) and **dynamic-output** (it terminates fusion and the C10 prefix and is exempt from the static peak-live bound). The executor hands the current op set to the child node's `Process` as its `RenderNodeContext.Input`, threading `OutputScale`, `MaxWorkingScale`, and — exactly like the node-graph render boundary — the shared `PipelineDiagnostics` and `RenderTargetPool`, so the work the child drives counts on the owning renderer (FR-017) and shares the pool (FR-006); the child derives its own working scale from the input ops' `EffectiveScale` (the C3.2 carried density). The returned ops flow onward; ownership of the inputs passes to the child (it disposes or returns each), and a throw inside the child releases them (C7). **Structural identity** = the child's `RenderNodeFactory.NodeType` + the child resource's **stable identity token** (a collision-free per-instance id, `FilterEffect.Resource.StructuralId`, not an object hash code) — a swapped child instance or node type recompiles the plan; the child's `Resource.Version` is a per-frame parameter (rebind), so an animated child param re-renders through the same plan without a recompile (SC-002). The wrapper node is created per frame, not persisted: the executor is stateless and reentrant, and the realistic child (`NodeGraphFilterEffectRenderNode`) rebuilds its inner processor every `Process` (its caches live on the persisted graph-model resources), so persisting the wrapper would save no cache. A **top-level** `NodeGraphFilterEffect` is unchanged — it still goes through its own `RenderNodeFactory`/`NodeGraphFilterEffectRenderNode` at the `GraphicsContext2D` level; the `CustomRenderNode` path is only for the effect **embedded** in a describable graph.
7. **Topology is structural**: split division counts, compute pass counts, and branch counts are part of the `StructuralKey` — changing them recompiles exactly once (FR-010); they are never treated as size-only parameters.

## C4. Scheduling & synchronization

1. Passes are topologically ordered; same-backend runs are contiguous when the DAG allows.
2. `SyncBefore` is set only where pass *i* reads a resource last written by the other backend. The executor performs: Skia→Vulkan boundary = surface flush + `PrepareForSampling`; Vulkan→Skia = `PrepareForRender`. The `FlushSyncs` counter counts **schedule-level backend transitions** and equals their number in the schedule (test-asserted) — these are the only *executor-owned* sync points. Individual draws inside a plan keep Skia's own per-draw flush semantics (`ImmediateCanvas` / `RenderTarget.PrepareForSampling` behave exactly as they did before 004); the executor neither adds nor elides them. Executor-owned flush elision inside a plan is a recorded follow-up, deferred by maintainer decision.
3. `GeometryPass` canvases are opened/closed by the executor; a session cannot outlive its pass.

## C5. Plan cache invalidation (exhaustive list)

A cached plan is reused iff **both** hold: equal `StructuralKey`, same graphics context (not device-lost). Anything else ⇒ recompile exactly once, old plan's pooled resources released. **Bounds, ROIs, buffer sizes, and the resolved working scale are per-frame resource-resolution inputs, never invalidation triggers** — a parameter change that inflates bounds (blur sigma, stroke pen) MUST hit the cache and only re-resolve sizes. A value that changes topology (split division count, compute pass count, branch count) is structural and MUST miss the cache exactly once (C3.6). Parameter-only changes MUST hit the cache (SC-002: `PlanCompilations == 1` over 100 animated frames — including bounds-animating parameters; `ProgramCreations == 0` after frame 1 given warm `ProgramCache`).

## C6. Fallback execution (no Vulkan)

Plans containing only Skia-backend passes execute unchanged on raster/SwiftShader contexts (fused SKSL runs through Skia's raster pipeline). `ComputePass` applies its declared fallback. CI (software rendering) MUST pass with the same golden thresholds (FR-014).

## C7. Failure semantics

Pool exhaustion / allocation failure during execution, uniformly for every pass kind: preview (`MaxWorkingScale` finite) → drop the pass output, continue, log; delivery (export) → throw `InvalidOperationException` (message parity with the pre-redesign `ThrowIfDeliveryAllocationFailure`). A thrown pass releases all acquired resources (no pool leaks; leak-asserted in tests).

This is a deliberate **normalization** of path-dependent legacy behavior — today `Flush` drops/throws, but `CustomFilterEffectContext.CreateTarget` returns an *empty* target whose `Open` then throws unconditionally even in preview. Tests MUST cover the replacement of each legacy path (fused, geometry-session, compute) under forced allocation failure.

## C8. Counter semantics (normative definitions)

| Counter | Increment rule |
|---|---|
| `GpuPasses` | each executed draw/dispatch of a `CompiledPass` (a fused group = 1; K compute iterations = K) |
| `TargetAllocations` | each fresh GPU target creation (pool miss or non-pooled) |
| `PoolAcquires` / `PoolMisses` | each pool acquire / acquire that allocated |
| `FullFrameMaterializations` | each bake of an upstream operation into a pooled buffer so a geometry/compute/split pass can sample it as a texture (the executor's input materialization); a fused/Skia-filter pass bakes straight into its output and does not count |
| `FlushSyncs` | each backend-transition sync pair (C4.2) |
| `PlanCompilations` / `ProgramCreations` | each graph compile / each `SKRuntimeEffect` (or Vulkan pipeline) construction |
| `PrefixCacheHits` | each frame the pass-prefix output cache is reused instead of re-executing the prefix (C10); the skipped passes' `GpuPasses` / `TargetAllocations` do not occur |
| `CompositeLayerSaves` | each composite branch drawn through a **full-canvas** `SaveLayer` (C9.5); zero for src-over composites, one per branch for blend modes that alter the destination under a transparent source |

Counters are per-renderer, render-thread, always-on. Every counter is **publicly readable** (`Snapshot()` and the get-only properties), but mutation — the counter increments and `Reset()` — is **engine-internal** (`internal set` accessors, `internal Reset()`, reachable by the engine and the test assemblies via `InternalsVisibleTo`). This keeps the observation-only contract enforceable when the live instance is handed to public authoring callbacks (`GeometrySession.Diagnostics` / `PassUniformContext.Diagnostics`): plugin code can read but never forge or reset a counter, so `IRenderer.Diagnostics` stays trustworthy. `Snapshot()`/`Reset()` are the test API. These definitions are binding — SC-001/002/003/005 are expressed in them.

**Brush-backed nested renders (FR-017).** When a migrated effect fills with a `DrawableBrush`/`ImageBrush` (`BlendEffect`, `FlatShadow`, `DisplacementMapEffect`, the `DisplacementMapTransform` map), the pass renders the brush content to produce its shader. The executor threads the pass's `PipelineDiagnostics` into that nested render (via `GeometrySession.Diagnostics` / `PassUniformContext.Diagnostics` → `BrushConstructor`), so any effect *inside* the brush's drawable counts on the owning renderer's counters and the brush's own composition buffers count `TargetAllocations` — the nested work is observable, not swallowed by a throwaway instance. The `RenderTargetPool` is deliberately **not** threaded into this path: a brush renders while its outer effect pass already holds that pass's declared pooled leases, so an extra lease here would exceed the plan's static peak-live bound (C3.1 / FR-007). These brush-composition buffers are therefore counted but non-pooled (they re-allocate per frame) — correctness of the FR-007 bound over per-frame reuse.

## C9. Invariant-run fold into an adjacent composite (normative)

A `FusedShaderPass` composed **entirely** of color-filter stages (`ColorFilterStage`; no `RuntimeShaderStage`) that **immediately precedes** a `CompositePass` folds into that composite: the compiler drops the fused pass and attaches its ordered color-filter factories to the composite (`CompositePass.InputColorFilters`), and the executor applies their composed `SKColorFilter` to **each** branch draw the composite performs.

1. **Correctness basis.** A composite draws each current branch exactly once (C4, fan-in). A color-filter run is coordinate-invariant (identity bounds, per-pixel), so applying it to each branch draw is byte-identical to first baking every branch through the run and then compositing — the same equivalence C2 already grants a `ColorFilterNode` adjacent to a Skia-filter run. The composed filter is applied **before** the blend (Skia's layer-paint color-filter semantics), i.e. `blend(colorFilter(branch), dst)`. The fold rides the per-branch layer the composite opens when one is needed (C9.5), adding no layers of its own.
2. **Applicability.** A run containing a `RuntimeShaderStage` (an SKSL snippet/whole-source shader) is **not** an `SKColorFilter` and does **not** fold — it stays its own pass. The fold only collapses a pure color-filter run that is *immediately* upstream of the composite; a non-invariant pass (Skia filter, geometry, compute) between the run and the composite blocks it. The composite's `BlendMode` must be **`SrcOver`**: a non-src-over composite draws each branch through a full-canvas `SaveLayer` (C9.5), and a color filter that affects transparent black riding that full-canvas layer would filter the transparent pixels *outside* the branch bounds too, diverging from the unfused plan whose fused intermediate is branch-bounded. Restricting the fold to src-over is the simplest sound rule (a per-filter transparent-affecting analysis would be needed to fold safely into a non-src-over layer, and is not worth it).
3. **Counters (C8).** The fold removes the fused pass's draw and its per-branch intermediate targets: `GpuPasses` and `TargetAllocations` **decrease** (never increase). The composite still counts one `GpuPass`. Structure-determined, so a fold is reproduced identically on a plan-cache hit (the folded factories are per-frame parameters re-extracted by `ParameterBlock`, so an animated fold amount rebinds without a recompile).
4. **Parity.** Because the fold is semantics-preserving, a scene's frozen parity reference (e.g. `chain-SplitTree`) MUST still pass **un-regenerated** after the fold lands.
5. **Branch draw path (normative).** How the executor draws each composite branch is blend-mode-dependent, because a full-canvas `SaveLayer` per branch costs one whole-target round trip per branch (measured: a 3×3 `SplitTree` composite alone was ~21 ms at 4K — ~9× the branch area in layer traffic — versus ~4-5 ms without the full-canvas layers):
   - **Src-over, no fold filter**: the branch draws directly — no layer, no blend push. Src-over preserves the destination wherever the source is transparent, so this is pixel-identical to the layered draw.
   - **Src-over with a fold filter**: the branch draws through a `SaveLayer` **bounded to the branch's bounds** carrying the composed `SKColorFilter`. Skia widens a bounded layer itself when the filter affects transparent black, so bounding never changes pixels; a branch-sized layer is also exactly the extent the pre-fold per-branch bake had, so the fold equivalence in (1) is preserved.
   - **Any other blend mode**: the branch draws through the full-canvas `SaveLayer` and counts one `CompositeLayerSaves` (C8). The extent is semantically load-bearing there — e.g. `Multiply` zeroes the destination wherever the source is transparent, so pixels outside the branch bounds must participate in the blend. A fold filter never rides this layer: the fold is src-over-only (C9.2), so a non-src-over composite always carries an empty `InputColorFilters`.
   `CompositeDrawPathTests` pins all three paths (src-over direct = 0 layer saves, folded src-over = 0, non-src-over = one per branch).

## C10. Pass-prefix output caching (normative)

Fusion required one render node per effect group (C1, SC-001), which made the whole group a single `RenderNodeCache` unit: any animated child marks the node changed every frame, so the outer node cache never engages and a heavy *static* prefix of the chain (a blur, a stroke) re-executes every frame even though only the tail animates — a granularity the pre-004 per-child flattening had. C10 restores it at **pass** granularity inside `PlanFilterEffectRenderNode` (the default node returned by `FilterEffect.Resource.CreateRenderNode()`), keeping one node per group.

1. **Provenance.** `EffectGraphBuilder` stamps each `EffectNode` with the top-level group-child index it was described under (`FilterEffectGroup.Describe` brackets each child; ungrouped/single effects and every node inside one non-group effect are child `0`; a nested group inherits its outer child index). The compiler propagates a provenance range `[ProvenanceMinChild, ProvenanceMaxChild]` onto each `CompiledPass` (a fused/Skia run spanning several children carries the span). This holds on plan-cache hits too (`ParameterBlock` rebuilds passes through the same grouping).

2. **Engagement conditions (all required).** The cache retains the output of the last **capturable linear** pass whose whole provenance range lies within the leading run of children each stable for at least `RenderNodeCache.Count` (3) consecutive frames. A pass is capturable only when it is a `FusedShaderPass` or `SkiaFilterPass` and is neither render-time-resolved nor dynamic-output; the prefix ends at the first non-capturable pass — **any `SplitPass`, `CompositePass`, `NestedGraphPass`, `CustomRenderNodePass`, geometry, compute, or dynamic-output pass terminates the prefix (v1 linear-prefix scope)**. Engagement additionally requires an unchanged `StructuralKey`, graphics context, resolved working scale, input-bounds signature (the bounds + supply density of the ops feeding the plan), and input-subtree stability (the same `RenderNodeCacheHelper.CanCacheRecursiveChildrenOnly` predicate the node cache uses, so a changed drawable below invalidates the prefix). The whole plan is **never** cached — the all-stable case belongs to the outer node cache, so at least the tail pass stays outside the prefix. Engagement is only attempted on the pooled render path (`RenderNodeContext.Pool != null`); the pool-less golden/frozen harnesses render once and never reach the threshold.

3. **Retention & resume.** On the engagement frame the plan runs in full and a shallow copy of the capture pass's pooled buffer is retained (its ref keeps the pooled lease from being reclaimed — `RenderTargetPool` returns a buffer only at its lease's last release). Ownership of that shallow-copy ref is single and explicit: the capture sink holds it during execution, and either the cache **adopts** it when the capture execution completes or the sink **releases** it if a pass after the capture pass throws — a thrown pass therefore frees the captured lease too (C7), never stranding it for finalizer-only reclaim. Subsequent stable frames execute the plan from pass `k+1` with the retained buffer as the input operation (correct bounds + `EffectiveScale` at the cached density); passes `0..k` neither draw nor allocate, and the fresh input ops are discarded (the stable prefix already encapsulates them). Because the plan is resolved with `Rect.Invalid` (full bounds, no ROI crop), the retained prefix is the full pass-`k` output, so it is always a superset of any tail request. Each reused frame counts one `PrefixCacheHits` (C8).

4. **Invalidation (exact — never replay a stale prefix).** The retained buffer is released back to the pool (its lease's last release returns it) on **any** of: a prefix child's per-child resource version change (its stability run drops at or below the retained pass's `ProvenanceMaxChild`), a `StructuralKey` change, a graphics-context change (device loss), a resolved-working-scale change, an input-bounds-signature change, input-subtree instability, **the outer `RenderNodeCache` engaging over this node or an ancestor** (below), or node dispose. After release the node re-warms from scratch (three stable frames) before re-engaging.

5. **Interplay & correctness.** The outer `RenderNodeCache` is unchanged in *when* it engages: when the whole node is stable it still wins (tile replay before `Process`), and a fully static group never engages C10 (the whole-plan guard declines it). Once the outer cache engages — at this node or at an ancestor whose subtree includes it — `Process` never runs on this node again, so its retained prefix lease would otherwise be stranded outside every pool budget (a leased buffer is invisible to `RenderTargetPool.Trim`'s idle eviction and its byte soft-cap, which both walk idle buckets only). To prevent that, `RenderNodeCacheHelper.CreateDefaultCache` notifies the whole cached subtree (`RenderNode.OnServedFromCache`) as the tiles are stored; `PlanFilterEffectRenderNode` overrides it to release the retained prefix. A later outer-cache invalidation re-runs `Process`, which re-warms the prefix from scratch. Single-frame renders (the SC-001 counter fixtures) never reach the engagement threshold, so those counters are unaffected. On a capture frame the retained buffer inflates the intra-frame peak by exactly one past the plan's declared bound, so the FR-007 peak-live DEBUG assertion is relaxed by a one-buffer allowance **keyed on an actual capture** (`captureSink.Captured`), so a capture pass skipped for an empty ROI keeps the bound exact; resume frames subtract the retained lease as the execution baseline and assert the unmodified bound. Prefix eligibility assumes strict A1 resource-only reads: an effect that reads state outside its `Resource` in `Describe` (a static/ambient/global) is not tracked by the per-child version signature and would stale-replay a resumed prefix — the in-tree effects all conform, and this is normative for plugin effects.
