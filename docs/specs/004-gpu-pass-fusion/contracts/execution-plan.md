# Contract: Graph Compilation & Execution

**Feature**: `004-gpu-pass-fusion` | Binding on: the compiler (`EffectGraph` → `CompiledPlan`) and `PlanExecutor`

## C1. Fusion rules (normative)

1. A **fusion group** is a maximal run of adjacent nodes on one edge-chain where every node is coordinate-invariant (`ShaderNode` snippet with `IsCoordinateInvariant`, or `ColorFilterNode`).
2. One fusion group compiles to **one `FusedShaderPass`**: one draw into one target, built by shader composition (input image shader → alternating `WithColorFilter` wraps and `SKRuntimeEffect` child-shader wraps, in node order). Node order is strictly preserved.
3. Adjacent SKSL snippets within a group SHOULD be merged into one generated program (uniforms prefixed `fe{N}_`, chained `apply` calls); merging is an optimization and MUST NOT change ordering or results beyond floating-point tolerance.
4. Fusion groups never cross: a non-invariant node, a `SplitNode`/`CompositeNode`, a backend transition, or a group that would exceed the composition budget. The compiler caps a pass at 16 shader stages and 224 four-component uniform vectors (the supported GLES 3 fragment-uniform floor), splitting before the next node would exceed either limit. One authoring node larger than the uniform floor remains a singleton because the compiler cannot split inside a node (FR-005).
5. A fusion group at the head of a chain samples the group's input exactly once per output pixel; a group's pass ROI equals the intersection of the requested output ROI with the group bounds (identity transforms).
   Production pulls carry the parent's logical request in `RenderNodeContext.RequestedBounds`; the default plan node passes it to `ResolveResources`, so the graph-output intersection and the internal backward walk both run on the production path. `Rect.Invalid` explicitly requests the full graph output.

## C2. Skia filter grouping

Adjacent `SkiaFilterNode`s compose into a single `SkiaFilterPass` (one filtered draw), reproducing the pre-redesign `SKImageFilterBuilder` accumulation (now removed) as an explicit plan pass. A `ColorFilterNode` adjacent to a Skia filter run MAY fold into the same paint (Skia applies color filters in the same draw) when node order allows; otherwise it joins the neighboring fusion group.

## C3. Resource planning

1. The compiled `ResourcePlan` declares the **structural shape** of every intermediate (`Id`, format, `[FirstUse, LastUse]`): each pass's outputs (one per current operation, read by the next pass) **and** its statically bounded pass-scoped scratch — the materialized input a geometry/compute/split pass bakes, the **source-halo bake buffer** of a non-invariant whole-source fused pass, and exactly the color scratch maximum declared by each compute descriptor. `IComputeContext` rejects an acquire beyond that maximum in release builds, so a plugin callback cannot exceed the compiled plan; debug builds additionally compare the pool's measured peak against the full declared interval bound. Concrete device sizes and per-pass ROIs are computed **every frame** by the resource-resolution pass (pure `Rect` math over the freshly described bounds) — parameter-driven bounds (animated blur sigma, stroke pen, split geometry) MUST flow through without recompilation.
   The backward-ROI walk is always active in production. A concrete parent request seeds the last pass with its intersection; `Rect.Invalid` seeds it from `OutputBounds`. Either path can propagate a **deflating** ROI to upstream intermediates (a fixed `Clipping`, `DropShadowOnly`). An author of a non-invariant pass whose uniforms are anchored in full-frame device space (a tile grid, a pivot, a cross-sampled child) MUST declare `RenderTime` or author ROI-relative so a cropped bake cannot disagree with its uniforms.
2. Device sizes obey the 003 budget with **legacy carry parity** (FR-012): the working scale is resolved once at the effect boundary (`ResolveWorkingScale`); resource resolution then re-clamps per buffer (`ClampWorkingScaleToBufferBudget`, 16 384 px/axis) in schedule order with a **monotonically non-increasing `w` carried along each chain** — a clamped materializing pass propagates its reduced `w` to downstream passes (the pre-redesign `FilterEffectActivator.Flush` mutated the activator's `WorkingScale`), while intra-pass allocations (ping-pong, geometry-session targets) re-clamp locally without affecting the carry (as the pre-redesign `CustomFilterEffectContext.CreateTarget` did). Same functions, same resulting densities as the pre-redesign pipeline. A **nested-graph branch** (research D8) is a materializing single-op path like any other: the executor describes, resolves, and recursively executes each branch at the **carried** working scale of the operation feeding it (`min(boundary w, the op's EffectiveScale)`; unbounded vector ops take the boundary scale) — never the raw outer working scale, which would re-raise a density an upstream clamped op already reduced.
3. **Uniform values AND density-/target-dependent child shaders are applied at execution, against the pass's resolved density and exact operation bounds.** When the executor builds a fused pass's shader (`PlanExecutor.BuildRuntimeRun`), it hands the same `PassUniformContext` — carrying the pass's actual `WorkingScale`, output buffer size (`TargetWidth`/`TargetHeight`), and exact logical `TargetBounds` for this operation (branch-local after fan-out), after any re-clamp/carry — to both every `UniformBinding` and every `ChildBinding`. A uniform binding whose device-space value depends on that density or target (`DensityScaledFloat2`, `Deferred`) computes it here; a `ChildBinding.Deferred` child (a cross-sampled displacement map) has its shader *product* built here from the same context, so a pass executing below its describe-time working scale or over one split branch gets a coherent offset/pivot/resolution AND a coherently scaled and sized map lookup (authoring contract A4). The deferred child product is executor-owned: it is tracked on the pass's disposable list and freed after the pass's draw, alongside the built runtime shader (an eager child stays graph-/caller-owned and is left alone). Structural fields (uniform/child names, shader source) are untouched — this is a per-frame parameter resolution, never a recompile.
4. Compute iteration declares an explicit maximum concurrent color scratch count. PixelSort declares two color targets regardless of iteration count; a one-pass script declares zero scratch. Every GLSL dispatch uses a color-only fullscreen render pass because depth testing and depth writes are disabled.
5. Empty resolved ROI ⇒ the executor skips the pass at runtime (spec edge case "zero-size targets"); the plan itself is unchanged. The two skip causes differ: an empty resolved **output** (a shrinking pass, e.g. a fully-closed clip) produces **no** downstream operation — the pass's input is released, matching the legacy `Clipping.Apply` target removal — while an empty **input** to an identity pass passes the already-empty operation through.
   **Stable branch identity:** every `ISplitEmitter.Emit` call receives an execution-local ordinal in authored Emit order before empty-bounds, allocation-failure, `DiscardOutput`, or degenerate-shrink handling. A surviving operation carries that ordinal through one-to-one descriptor passes even when a sibling produces no output. `CompositePass.InputOffsets` and `NestedGraphPass` branch indices use the stable ordinal, never the compressed position in the live-operation list; an offset array shorter than the ordinal still defaults that branch to zero offset. Composite fan-in ends the ordinal scope, and a later split starts a new one. Nested-plan cache entries are keyed and pruned by the live stable-ordinal set, so dropping branch 1 cannot make branch 2 reuse branch 1's plan or delayed clock.
   - **Render-time output tightening.** A `RenderTime` geometry pass is sized to its full input bounds (the resolver cannot resolve its output forward), so a pass that narrows its content only at render time — `Clipping.AutoClip`, whose margins are detected by sampling the input — calls `GeometrySession.SetOutputBounds(Rect)` inside the callback to emit a **sub-rect** of the allocated buffer. The executor copies that sub-rect into a tighter pooled output target, so the downstream operation's bounds and hit-test region match the content (the legacy `Clipping.Apply` sized its output target to the detected clip rect). Counters: the extra pooled acquire counts one `TargetAllocations`/`PoolAcquires` (C8); the geometry pass still counts one `GpuPass` (the blit is part of emitting the pass output, not a new pass). Peak-live: the executor acquires the tighter target and blits **while the full output is still leased**, then releases the full output — this reuses the concurrency slot the materialized input held during render (input-scratch is already released by then), so the pass's concurrent-lease peak is unchanged and the ResourcePlan `GeometryPass` declaration (§C3.1) is not widened. The requested rect MUST be a shrink (contained in the pass bounds); a grow throws, and a degenerate (empty) request drops the output like `DiscardOutput`, which supersedes a shrink regardless of order. A **split branch** honors `SetOutputBounds` the same way — `SplitEmitter` blits the requested sub-rect into a tighter pooled target and publishes the tightened branch bounds (discard still supersedes); the split is a dynamic-output pass (exempt from the static peak-live bound), and the transient tight lease is acquired while the branch target is still leased. An `AutoClip` combined with **`AutoCenter`** publishes the centered `TargetBounds` (re-centering the kept window inside the input frame, per the legacy `Clipping.Apply`) and shifts the drawn kept region onto it, falling back to the un-centered `NewBounds` when a downstream-narrowed buffer cannot contain the centered rect (`SetOutputBounds` requires containment). **Downstream propagation:** the engage condition fires whenever the actual op **differs** from the resolver's expected input (the previous pass's resolved output ROI), and the three cases diverge by how it differs. For a **true shrink** (`expectedInput.Contains(op.Bounds)` — an upstream AutoClip tightened it), a later linear non-invariant pass re-derives its bake window as `resolvedRoi ∩ ForwardBounds(op.Bounds)` — shrink-only, the resolve-time ROI stays the upper bound — so the tightening survives through bounds-inflating successors (an AutoClip → DropShadow publishes bounds derived from the tightened rect, not the frame-start resolution). For a **shifted** op (a dynamic `CustomRenderNode`/`NestedGraph` predecessor that translates its output so the op escapes the expected input) **or a grown one** (that same predecessor inflates its output so `op.Bounds ⊇ expectedInput`), the resolve-time ROI — computed from the pre-change describe-time bounds — is stale for this frame, so the pass maps the actual op forward directly (`ForwardBounds(op.Bounds)`, like a fan-out branch) rather than intersecting with the stale ROI (which would clip or empty a shifted op and crop a grown one). When the op matches the expectation the resolved ROI is kept verbatim: a forward map is authored against the pass's semantic input frame (a fixed `Clipping` deflates whatever rect it receives), so applying it to a merely ROI-narrowed op would double-apply the crop.
6. **Dynamic outputs**: a pass declared `IsDynamicOutputs` (execution-time-resolved output count, e.g. contour-based part splitting, a nested-graph branch, or an `CustomRenderNodePass`) has no static output decls; the executor allocates its outputs from the pool at execution time, releases them within the frame (leak-asserted), and counts them. The FR-007 static peak-live bound covers declared intermediates only.
   - **Custom-render-node pass (`CustomRenderNodePass`)**: the declarative home for an effect whose execution lives in a custom `FilterEffectRenderNode` (the canonical case is `NodeGraphFilterEffect`). Every container routes children through `EffectGraphBuilder.Effect(FilterEffect.Resource)`: it reads `RenderNodeFactory` exactly once, describes a default-plan child normally, or captures the same non-default sealed factory in `CustomRenderNodeDescriptor` and then `CustomRenderNodePass`. This makes behavior placement-independent even for an older direct `FilterEffect` subclass with a factory override; the executor never re-reads a stateful virtual getter. `CustomRenderNodeFilterEffect` remains the recommended base because its abstract resource makes the factory mandatory at compile time and its sealed `Describe` calls `Effect`. Semantics: **render-time resolved** and **dynamic-output**, terminating fusion and C10 and exempting the pass from the static peak-live bound. The executor hands the current op set to the child node's `Process`, threading scales, cache policy, shared diagnostics, and pool. The opaque child receives `RequestedBounds = Rect.Invalid`; returned operations retain the wrapper node until their final disposal. **Structural identity** is the captured factory's `NodeType` plus the resource's collision-free `StructuralId`; fresh per-frame passes rebind the current captured factory/resource while a node-type or resource swap recompiles. A top-level custom effect still runs through the same factory at `GraphicsContext2D` level.
7. **Declared topology is structural**: compute pass counts and a `SplitNodeDescriptor.Static` branch count are part of the `StructuralKey` — changing them recompiles exactly once (FR-010). `SplitNodeDescriptor.Dynamic` is the explicit exception: its output count is resolved and leak-checked at execution and is not a key part. The built-in `SplitEffect` chooses static only for a first-level grid with ≥1×1 logical tiles and ≤4096 branches; sub-pixel grids, splits after fan-out, and larger grids remain dynamic.

## C4. Scheduling & synchronization

1. Passes are topologically ordered; same-backend runs are contiguous when the DAG allows.
2. `SyncBefore` is set where a pass must move the previous output back to the backend on which that pass starts consuming it. A compute pass first materializes through Skia and then performs its own Skia→Vulkan `PrepareForSampling`, so Compute→Compute pays and counts Vulkan→Skia→Vulkan rather than being treated as a same-backend edge. Closing an `ImmediateCanvas` and drawing/snapshotting a Skia surface into another Skia canvas never flushes. A geometry/split/CPU-fallback callback that needs CPU pixels declares `requiresReadback`; the executor synchronizes the input once before invoking it, and `EffectInput.Snapshot()` rejects undeclared readback. `FlushSyncs` counts every plan-owned backend transition plus every declared CPU-readback synchronization, matching the actual synchronous boundaries paid by effect execution.
3. `GeometryPass` canvases are opened/closed by the executor; a session cannot outlive its pass.

## C5. Plan cache invalidation (exhaustive list)

A cached plan is reused iff **both** hold: equal `StructuralKey`, same graphics context (not device-lost). Anything else ⇒ recompile exactly once, old plan's pooled resources released. **Bounds, ROIs, buffer sizes, and the resolved working scale are per-frame resource-resolution inputs, never invalidation triggers** — a parameter change that inflates bounds (blur sigma, stroke pen) MUST hit the cache and only re-resolve sizes. A compute pass-count or static split branch-count change is structural and MUST miss exactly once; a dynamic split's execution-time count is a parameter and MUST hit while the descriptor remains dynamic (C3.6). Parameter-only changes MUST hit the cache (SC-002: `PlanCompilations == 1` over 100 animated frames — including bounds-animating parameters; `ProgramCreations == 0` after frame 1 given warm `ProgramCache`). A `NestedGraphPass` follows the same rule through an internal hierarchical cache: each persistent parent node owns cache entries keyed by nested-node ordinal and the stable runtime branch ordinal from C3.5, and each entry owns its child hierarchy. Unvisited node ordinals and branch ordinals absent from the current live set are pruned, so removed topology cannot retain plans or descendant caches.

## C6. Fallback execution (no Vulkan)

Plans containing only Skia-backend passes execute unchanged on raster/SwiftShader contexts (fused SKSL runs through Skia's raster pipeline). `ComputePass` applies its declared fallback. CI (software rendering) MUST pass with the same golden thresholds (FR-014).

## C7. Failure semantics

Pool exhaustion / allocation failure during execution, uniformly for every pass kind: `RenderIntent.Preview` → drop the pass output, continue, log; `RenderIntent.Delivery` → throw `InvalidOperationException` (message parity with the pre-redesign `ThrowIfDeliveryAllocationFailure`). `RenderIntent` is propagated independently from `MaxWorkingScale`, so adding a finite delivery quality cap cannot change failure policy. A thrown pass releases all acquired resources (no pool leaks; leak-asserted in tests).

An ordinary exception thrown by a compute dispatch callback propagates after cleanup by default; this policy is separate from the no-Vulkan fallback. `ComputeDispatchFailureBehavior.IdentityInPreview` is an explicit compatibility policy for effects such as PixelSort: `RenderIntent.Preview` keeps the source operation, while delivery and cancellation still throw. Dispatch-count, terminal-copy, and color-scratch violations are authoring-contract failures and always throw; a scratch allocation failure remains governed by the preview-drop/delivery-throw rule above. Static split callbacks likewise throw before an excess branch allocation and after a normal return with fewer emitted branches than declared.

**GPU teardown drain (batched).** Destroying a Vulkan-backed pooled buffer first attempts to complete every recorded-but-unflushed Skia op that still references its image. The pool defers its soft byte-cap eviction from individual lease returns to the next frame-boundary `Trim`, where idle and byte-cap eviction share one `GpuDisposeBatch` drain. Teardown is batched the same way. If a context-loss drain throws, `VulkanTexture2D.Dispose` still best-effort releases the image view, image, and device memory and does not throw; the disposed guard cannot convert that failure into a permanent native-handle leak.

This is a deliberate **normalization** of path-dependent legacy behavior — today `Flush` drops/throws, but `CustomFilterEffectContext.CreateTarget` returns an *empty* target whose `Open` then throws unconditionally even in preview. Tests MUST cover the replacement of each legacy path (fused, geometry-session, compute) under forced allocation failure.

## C8. Counter semantics (normative definitions)

| Counter | Increment rule |
|---|---|
| `GpuPasses` | each executed draw/dispatch of a `CompiledPass` (a fused group = 1; K compute iterations = K) |
| `TargetAllocations` | each fresh GPU target creation (pool miss or non-pooled) |
| `PoolAcquires` / `PoolMisses` | each pool acquire / acquire that allocated |
| `FullFrameMaterializations` | each bake of an upstream operation into a pooled buffer so a geometry/compute/split pass can sample it as a texture (the executor's input materialization); a fused/Skia-filter pass bakes straight into its output and does not count |
| `FlushSyncs` | each backend-transition sync pair plus each descriptor-declared CPU-readback synchronization (C4.2) |
| `PlanCompilations` / `ProgramCreations` | each graph compile / each `SKRuntimeEffect` (or Vulkan pipeline) construction |
| `PrefixCacheHits` | each frame the pass-prefix output cache is reused instead of re-executing the prefix (C10); the skipped passes' `GpuPasses` / `TargetAllocations` do not occur |
| `CompositeLayerSaves` | each composite branch drawn through a **full-canvas** `SaveLayer` (C9.5); zero for src-over composites, one per branch for blend modes that alter the destination under a transparent source |

Counters are per-renderer, render-thread, always-on. Every counter is **publicly readable** (`Snapshot()` and the get-only properties), but mutation — the counter increments and `Reset()` — is **engine-internal** (`internal set` accessors, `internal Reset()`, reachable by the engine and the test assemblies via `InternalsVisibleTo`). This keeps the observation-only contract enforceable when the live instance is handed to public authoring callbacks (`GeometrySession.Diagnostics` / `PassUniformContext.Diagnostics`): plugin code can read but never forge or reset a counter, so `IRenderer.Diagnostics` stays trustworthy. `Snapshot()` is the public observation API; `Reset()` is only an internal test hook. These definitions are binding — SC-001/002/003/005 are expressed in them.

**Brush-backed nested renders (FR-017).** When a migrated effect fills with a `DrawableBrush`/`ImageBrush` (`BlendEffect`, `FlatShadow`, `DisplacementMapEffect`, the `DisplacementMapTransform` map), the pass renders the brush content to produce its shader. The executor threads the pass's `PipelineDiagnostics` into that nested render (via `GeometrySession.Diagnostics` / `PassUniformContext.Diagnostics` → `BrushConstructor`), so any effect *inside* the brush's drawable counts on the owning renderer's counters and the brush's own composition buffers count `TargetAllocations` — the nested work is observable, not swallowed by a throwaway instance. The `RenderTargetPool` is deliberately **not** threaded into this path: a brush renders while its outer effect pass already holds that pass's declared pooled leases, so an extra lease here would exceed the plan's static peak-live bound (C3.1 / FR-007). These brush-composition buffers are therefore counted but non-pooled (they re-allocate per frame) — correctness of the FR-007 bound over per-frame reuse.

## C9. Invariant-run fold into an adjacent composite (normative)

A `FusedShaderPass` composed **entirely** of color-filter stages (`ColorFilterStage`; no `RuntimeShaderStage`) that **immediately precedes** a `CompositePass` folds into that composite: the compiler drops the fused pass and attaches its ordered color-filter factories to the composite (`CompositePass.InputColorFilters`), and the executor applies their composed `SKColorFilter` to **each** branch draw the composite performs.

1. **Correctness basis.** A composite draws each current branch exactly once (C4, fan-in). A color-filter run is coordinate-invariant (identity bounds, per-pixel), so applying it to each branch draw is byte-identical to first baking every branch through the run and then compositing **when the branch is vector/unbounded or its concrete `EffectiveScale` equals the composite target density**. Skia paint filters run after texture interpolation, so the equivalence does not hold across density-changing resampling. The composed filter is applied **before** the blend (Skia's layer-paint color-filter semantics), i.e. `blend(colorFilter(branch), dst)`. The fold rides the per-branch layer the composite opens when one is needed (C9.5), adding no layers of its own.
2. **Applicability.** A run containing a `RuntimeShaderStage` (an SKSL snippet/whole-source shader) is **not** an `SKColorFilter` and does **not** fold — it stays its own pass. The fold only collapses a pure color-filter run that is *immediately* upstream of the composite; a non-invariant pass (Skia filter, geometry, compute) between the run and the composite blocks it. The composite's `BlendMode` must be **`SrcOver`**: a non-src-over composite draws each branch through a full-canvas `SaveLayer` (C9.5), and a color filter that affects transparent black riding that full-canvas layer would filter the transparent pixels *outside* the branch bounds too, diverging from the unfused plan whose fused intermediate is branch-bounded. The compiler preserves the original fused pass as `InputColorFilterFallback`; if any concrete branch density differs from the resolved composite density, the executor runs that pass at every branch's carried density before compositing/resampling instead of using the paint fold.
3. **Counters (C8).** On the density-compatible fast path, the fold removes the fused pass's draw and its per-branch intermediate targets: `GpuPasses` and `TargetAllocations` decrease. On a density-mismatch fallback, those per-branch draws and allocations are restored, matching the unfused plan, while the composite still counts one `GpuPass`. Structure and the runtime carried densities determine the path; plan-cache hits re-extract both the folded factories and preserved fallback pass, so animated parameters rebind without a recompile.
4. **Parity.** Because the fold is semantics-preserving, a scene's frozen parity reference (e.g. `chain-SplitTree`) MUST still pass **un-regenerated** after the fold lands.
5. **Branch draw path (normative).** How the executor draws each composite branch is blend-mode-dependent, because a full-canvas `SaveLayer` per branch costs one whole-target round trip per branch (measured: a 3×3 `SplitTree` composite alone was ~21 ms at 4K — ~9× the branch area in layer traffic — versus ~4-5 ms without the full-canvas layers):
   - **Src-over, no fold filter**: the branch draws directly — no layer, no blend push. Src-over preserves the destination wherever the source is transparent, so this is pixel-identical to the layered draw.
   - **Src-over with a fold filter**: the branch draws through a `SaveLayer` **bounded to the branch's bounds** carrying the composed `SKColorFilter`. Skia widens a bounded layer itself when the filter affects transparent black, so bounding never changes pixels; a branch-sized layer is also exactly the extent the pre-fold per-branch bake had, so the fold equivalence in (1) is preserved.
   - **Any other blend mode**: the branch draws through the full-canvas `SaveLayer` and counts one `CompositeLayerSaves` (C8). The extent is semantically load-bearing there — e.g. `Multiply` zeroes the destination wherever the source is transparent, so pixels outside the branch bounds must participate in the blend. A fold filter never rides this layer: the fold is src-over-only (C9.2), so a non-src-over composite always carries an empty `InputColorFilters`.
   `CompositeDrawPathTests` pins all three paths (src-over direct = 0 layer saves, folded src-over = 0, non-src-over = one per branch).

## C10. Pass-prefix output caching (normative)

Fusion required one render node per effect group (C1, SC-001), which made the whole group a single `RenderNodeCache` unit: any animated child marks the node changed every frame, so the outer node cache never engages and a heavy *static* prefix of the chain (a blur, a stroke) re-executes every frame even though only the tail animates — a granularity the pre-004 per-child flattening had. C10 restores it at **pass** granularity inside `PlanFilterEffectRenderNode` (the internal default node selected by `FilterEffect.Resource.RenderNodeFactory`), keeping one node per group.

1. **Provenance.** `EffectGraphBuilder` stamps each `EffectNode` with the top-level group-child index it was described under (`FilterEffectGroup.Describe` brackets each child; ungrouped/single effects and every node inside one non-group effect are child `0`; a nested group inherits its outer child index). The compiler propagates a provenance range `[ProvenanceMinChild, ProvenanceMaxChild]` onto each `CompiledPass` (a fused/Skia run spanning several children carries the span). This holds on plan-cache hits too (`ParameterBlock` rebuilds passes through the same grouping).

2. **Engagement conditions (all required).** The cache retains the output of the last **capturable linear** pass whose whole provenance range lies within the leading run of children each stable for at least `RenderNodeCache.Count` (3) consecutive frames. A pass is capturable only when it is a `FusedShaderPass` or `SkiaFilterPass` and is neither render-time-resolved nor dynamic-output; the prefix ends at the first non-capturable pass — **any `SplitPass`, `CompositePass`, `NestedGraphPass`, `CustomRenderNodePass`, geometry, compute, or dynamic-output pass terminates the prefix (v1 linear-prefix scope)**. Engagement additionally requires an unchanged `StructuralKey`, graphics context, resolved working scale, input-bounds signature (the bounds + supply density of the ops feeding the plan), and input-subtree stability (the same `RenderNodeCacheHelper.CanCacheRecursiveChildrenOnly` predicate the node cache uses, so a changed drawable below invalidates the prefix). The whole plan is **never** cached — the all-stable case belongs to the outer node cache, so at least the tail pass stays outside the prefix. Engagement is only attempted on the pooled render path (`RenderNodeContext.Pool != null`); the pool-less golden/frozen harnesses render once and never reach the threshold.

3. **Retention, renderer budget & resume.** On the engagement frame the plan runs and a shallow copy of the capture pass's pooled buffer is retained (its ref keeps the pooled lease from being reclaimed — `RenderTargetPool` returns a buffer only at its lease's last release). Every renderer pool owns a separate retained-prefix byte budget (`DefaultMaxRetainedPrefixBytes`, 256 MiB) and LRU spanning **all** `PlanFilterEffectRenderNode` instances that share the pool. Capturing registers the exact RGBA16F buffer bytes; a hit touches the entry; crossing the cap evicts the least-recently-used node prefix and returns its lease. An individual buffer larger than the cap is not retained. `RetainedPrefixBytes` / `RetainedPrefixCount` expose the bounded state for diagnostics/tests. Ownership of the shallow-copy ref is single and explicit: the capture sink holds it during execution, and either the cache **adopts** it when the capture execution completes or the sink **releases** it if a pass after the capture pass throws — a thrown pass therefore frees the captured lease too (C7), never stranding it for finalizer-only reclaim. Subsequent stable frames execute the plan from pass `k+1` with the retained buffer as the input operation; passes `0..k` neither draw nor allocate. The cache records the resolved ROI/size/scale slice through the capture pass and reuses it only when the current parent request resolves to the exact same slice. Each reused frame counts one `PrefixCacheHits` (C8).

4. **Invalidation (exact — never replay a stale prefix).** The retained buffer is released back to the pool (its lease's last release returns it) on **any** of: a prefix child's per-child resource version change (its stability run drops at or below the retained pass's `ProvenanceMaxChild`), a `StructuralKey` change, a graphics-context change (device loss), a resolved-working-scale change, an input-bounds-signature change, input-subtree instability, **the outer `RenderNodeCache` engaging over this node or an ancestor** (below), or node dispose. After release the node re-warms from scratch (three stable frames) before re-engaging. Auxiliary pulls used for hit-testing and boundary calculation execute with their own requested ROI but neither prepare, resume, capture, release, nor otherwise mutate this frame-render prefix state; this prevents alternating full-bounds/cropped pulls from evicting each other.

5. **Interplay & correctness.** The outer `RenderNodeCache` is unchanged in *when* it engages: when the whole node is stable it still wins (tile replay before `Process`), and a fully static group never engages C10. A retained prefix is a live lease and therefore invisible to the pool's **idle** LRU, but it remains bounded by the renderer-wide retained-prefix LRU above. When the outer cache engages, `RenderNodeCacheHelper.CreateDefaultCache` also notifies the cached subtree (`RenderNode.OnServedFromCache`); `PlanFilterEffectRenderNode` releases its redundant retained entry immediately rather than consuming budget until eviction. Pool disposal clears every retained entry before idle teardown and continues disposal after individual failures. A later outer-cache invalidation re-runs `Process`, which re-warms the prefix from scratch. Single-frame renders never reach the engagement threshold. On a capture frame the retained buffer inflates the intra-frame peak by exactly one past the plan's declared bound, so the FR-007 peak-live DEBUG assertion is relaxed by one only when a capture occurred; resume frames subtract the retained lease as the execution baseline. Prefix eligibility assumes strict A1 resource-only reads: an effect that reads ambient state outside its `Resource` is not tracked and violates the authoring contract.
